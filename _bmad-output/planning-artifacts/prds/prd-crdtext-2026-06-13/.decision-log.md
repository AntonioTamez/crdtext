# Decision Log — CRDText PRD

---

## DL-001 — Propósito dual del PRD
**Decisión:** El PRD sirve tanto como guía de desarrollo como pieza de portafolio.
**Razón:** Confirmado por el usuario.
**Fecha:** 2026-06-13

## DL-002 — Modo de trabajo: Coaching path, Vision + Features
**Decisión:** Coaching path con entrada Vision + Features.
**Razón:** El usuario eligió explícitamente ambos.
**Fecha:** 2026-06-13

## DL-003 — Texto plano en MVP
**Decisión:** El editor soporta solo texto plano (insert/delete). Sin formato enriquecido.
**Razón:** Mantiene el foco en el algoritmo CRDT como diferenciador técnico, no en la UI del editor.
**Fecha:** 2026-06-13

## DL-004 — Identidad de usuarios automática
**Decisión:** Nombres y colores asignados automáticamente al conectarse. Sin autenticación.
**Razón:** Elimina fricción de onboarding y enfoca la atención en la sincronización.
**Fecha:** 2026-06-13

## DL-005 — N usuarios en arquitectura, 2 en demo
**Decisión:** El código soporta N usuarios concurrentes; el demo se presenta con 2.
**Razón:** Logoot/LSEQ soporta N sitios por diseño — implementarlo correctamente no cuesta más. Presentar como "arquitectura N usuarios" fortalece el argumento técnico en entrevistas sin complicar el demo.
**Fecha:** 2026-06-13

## DL-006 — Conflict Visualizer como log de operaciones en tiempo real
**Decisión:** El visualizador muestra un log de operaciones en tiempo real (sitio, tipo, posición, timestamp lógico).
**Razón:** Es el formato más claro para demostrar el funcionamiento del algoritmo a un entrevistador técnico.
**Fecha:** 2026-06-13

---

## DL-007 — Conflict Visualizer muestra identificadores internos Logoot/LSEQ
**Decisión:** El log muestra los identificadores completos `(site, clock, frac_position)` del árbol interno, no solo operaciones de nivel usuario.
**Razón:** La audiencia es técnica. Mostrar los identificadores reales habilita conversaciones de entrevista sobre el algoritmo directamente desde el demo.
**Fecha:** 2026-06-13

## DL-009 — Service workers incluidos en F3 (Offline-first)
**Decisión:** FR-3.7 agrega service worker para gestionar el caché del app shell y coordinar con IndexedDB.
**Razón:** `proyecto.md` los mencionaba explícitamente. Demuestran "Frontend avanzado" como señal técnica para portafolio.
**Fecha:** 2026-06-13

## DL-008 — Redis como pub/sub + log de operaciones
**Decisión:** Redis actúa como broker pub/sub y mantiene un log persistente de operaciones. Los clientes nuevos reproducen el log para reconstruir el estado. No se serializa el árbol CRDT completo en Redis.
**Razón:** Demuestra comprensión de event sourcing sin agregar complejidad innecesaria a escala de demo. La reconstrucción desde el log es técnicamente más interesante que deserializar un snapshot.
**Fecha:** 2026-06-13

## DL-010 — PRD finalizado
**Decisión:** PRD marcado como `status: final`.
**Fecha:** 2026-06-13
